# lambada 2.9.0

## scanFilesPath/scanFilesListing return real errors, not a bare ok/nil

`scanFilesPath` now returns `(path string, err error)` instead of
`(path string, ok bool)`. An unresolvable filename, a missing file, or a
directory all collapse to `os.ErrNotExist`; anything else -- permission
denied, a read-only filesystem, whatever else -- is logged with the
failing path and returned as-is. `scanFilesListing` logs its own `Glob`
error before returning it, and now always returns a non-nil `scans`
slice (allocated right after `Glob`, before the error check), so callers
get the same shape on success or failure.

`main.go`'s routes react accordingly. `GET /{$}` and `GET /files.json`
discard `scanFilesListing`'s error and render an empty listing -- the
failure is already logged at the source, and both the template's
`{{if not .Listing}}` and `toScansJSON`'s JSON output already treat nil
and empty the same way, so nothing downstream needed to change.
`GET`/`DELETE /download/{filename}` now distinguish 404
(`os.ErrNotExist`) from 500 (a real stat failure) via `errors.Is`, using
an `if`/`else if`/`else` chain instead of early returns. `DELETE`'s own
`os.Remove` failure still gets its own log line and 500.

This intentionally crosses a boundary 2.8.0 drew on purpose:
`scanfiles.go` had been made deliberately free of `log`/`net/http`
imports, matching `scandalous/lib/scan_files.rb`'s pure-filesystem-work
boundary. That tradeoff assumed the only failure mode worth handling was
"file not found." Real filesystem errors -- ownership or permission
trouble -- turned out to be worth logging at the source rather than
silently folding into the same "not found" bucket, so `log` is back in
`scanfiles.go`.

## New request logging

`middleware.go` adds `withLogging`, which wraps the whole mux and logs
one line per request (method, path, status, duration) -- the `net/http`
equivalent of what Puma gives `scandalous-web`'s access logging for
free. Wired in as `newServer(listenAddr, withLogging(newMux()))` in
`main()`.

## Test coverage

`scanFilesPath` gained direct unit tests (existing file, `os.ErrNotExist`
for a missing file/directory/invalid name, and a real permission-denied
case via `chmod 0000`, skipped when running as root), plus a
malformed-glob-pattern case for `scanFilesListing`. `main_test.go`
gained HTTP-level 500 assertions for `download`/`delete` under the same
permission-denied setup, and `middleware_test.go` covers `withLogging`.

`docs/FRAMEWORK.md` was also trimmed this cycle (editorial only -- no
research conclusions changed).

## Upgrading

No dependency changes, no wire-format changes -- `/files.json`'s shape
is untouched. The only behavior a client could actually observe:
a permission-denied `download`/`delete` now returns 500 instead of a
misleading 404, and a directory-listing failure that used to 500 now
renders an empty listing (logged server-side) instead of failing the
request outright.

Confirmed via `npm run check` and CI on real hardware.
