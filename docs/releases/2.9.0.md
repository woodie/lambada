# lambada 2.9.0

## scanFilesPath/scanFilesListing return real errors, not a bare ok/nil

`scanFilesPath` now returns `(path string, err error)` instead of
`(path string, ok bool)`. An unresolvable filename, a missing file, or a
directory all collapse to `os.ErrNotExist`; anything else -- permission
denied, a read-only filesystem, whatever else -- is logged with the
failing path and returned as-is. `scanFilesListing` logs its own `Glob`
error before returning it, and now always returns a non-nil `scans`
slice (allocated right after `Glob`, before the error check), so callers
get the same shape on success or failure.

`main.go`'s routes react accordingly. `GET /{$}` and `GET /files.json`
discard `scanFilesListing`'s error and render an empty listing -- the
failure is already logged at the source, and both the template's
`{{if not .Listing}}` and `toScansJSON`'s JSON output already treat nil
and empty the same way, so nothing downstream needed to change.
`GET`/`DELETE /download/{filename}` now distinguish 404
(`os.ErrNotExist`) from 500 (a real stat failure) via `errors.Is`, using
an `if`/`else if`/`else` chain instead of early returns. `DELETE`'s own
`os.Remove` failure still gets its own log line and 500.

This intentionally crosses a boundary 2.8.0 drew on purpose:
`scanfiles.go` had been made deliberately free of `log`/`net/http`
imports, matching `scandalous/lib/scan_files.rb`'s pure-filesystem-work
boundary. That tradeoff assumed the only failure mode worth handling was
"file not found." Real filesystem errors -- ownership or permission
trouble -- turned out to be worth logging at the source rather than
silently folding into the same "not found" bucket, so `log` is back in
`scanfiles.go`.

## New request logging

`middleware.go` adds `withLogging`, which wraps the whole mux and logs
one line per request (method, path, status, duration) -- the `net/http`
equivalent of what Puma gives `scandalous-web`'s access logging for
free. Wired in as `newServer(listenAddr, withLogging(newMux()))` in
`main()`.

## Test coverage

`scanFilesPath` gained direct unit tests (existing file, `os.ErrNotExist`
for a missing file/directory/invalid name, and a real permission-denied
case via `chmod 0000`, skipped when running as root), plus a
malformed-glob-pattern case for `scanFilesListing`. `main_test.go`
gained HTTP-level 500 assertions for `download`/`delete` under the same
permission-denied setup, and `middleware_test.go` covers `withLogging`.

## `DELETE`'s own failure folded into the same condition, no longer logged

`GET`/`DELETE /download/{filename}` moved from a `switch` to an
`if`/`else if`/`else` chain (fewer explicit `return`s). For `DELETE`
specifically, the `os.Remove` failure case is now folded directly into
the same condition as the earlier stat-error check --
`err != nil || os.Remove(path) != nil` -- relying on `||`'s short-circuit
to guarantee `os.Remove` is only ever called with a real path (never
when `err != nil` already holds a stat failure and `path` is still
`""`). The dedicated `log.Printf("delete error: %v", err)` line for a
failed `os.Remove` is gone -- `lambada-mta` owns every file this app
ever deletes (its own 24-hour cleanup runs unattended too, with no one
watching those log lines either), so a delete failing here is expected
to be as rare as it is uninformative to chase down without more context
than a bare error would give anyway.

## File reorganization: one file per concern

Three moves, no behavior change, all following the same principle
`scanfiles.go`/`server.go`/`middleware.go` already established --
config and logic live next to whichever file actually depends on them,
not lumped into `main.go` by default:

- `scanDir` moved from `main.go` to `scanfiles.go`, next to
  `scanFilesPath`, its only direct consumer -- mirrors
  `scandalous/lib/scan_files.rb`'s `ScanFiles.scan_folder`, which is
  likewise declared inside `ScanFiles` itself, not `web.rb`.
- `listenAddr` moved from `main.go` to `server.go`, next to `newServer`
  -- a looser parallel to Puma's `config/puma.rb` owning the bind
  address separately from `web.rb`'s routes, though `newServer` takes
  `addr` as a parameter rather than reading a global the way
  `scanFilesPath` reads `scanDir`.
- The embedded `static/` handling (`staticFS`, the `//go:embed static`
  directive, and the handler built from it) moved out of `main.go`
  entirely into a new `static.go`. No Sinatra precedent here --
  `scandalous/public/` is served automatically by the framework itself,
  no hand-written Ruby equivalent exists -- this one's grounded in the
  same-file-per-concern pattern alone. Unlike the rest of this repo,
  `static.go` carries zero lambada-specific logic (a pure
  `embed.FS` -> `http.Handler` conversion), so it's realistically
  copy-pasteable into a future Go web project with only the literal
  `"static"` string and the doc comments needing adjustment.

`main.go` is left with `newMux`, `init`, and `main` -- wiring and the
entry point, nothing else.

## `loglevel` package removed; each binary inlines its own one-liner

The shared `lambada/loglevel` package (`Apply()`, gated on
`LOG_LEVEL == "OFF"`) is gone. In its place, `cmd/lambada-web/main.go`
and `cmd/lambada-mta/main.go` each inline the same two lines directly
in their own `init()`:

```go
func init() {
	if os.Getenv("LOG_QUIET") != "" { log.SetOutput(io.Discard) }
}
```

This went through two intermediate shapes this session before landing
here -- first a duplicated `logging.go`/`silenceLogging()` per binary
(with its own `logging_test.go`), then, once that still felt like
ceremony for two lines that have only ever changed as same-shape env-var
renames (`LAMBADA_QUIET` -> `LOG_LEVEL` -> `LOG_QUIET`), inlined
directly into `init()` with the test coverage dropped deliberately --
`init()` can't be re-invoked per test case the way a standalone function
could, and the alternative (spawning a subprocess per `LOG_QUIET` value)
would be more ceremony in the other direction. Unlike `internal/envutil`
(kept as a real shared package -- see below), this one has no second
distinct call site to keep in sync; both binaries want the exact same
two lines, not the same function called with different arguments.

Also renamed in the same pass: `LOG_LEVEL=OFF` -> `LOG_QUIET` (any
non-empty value, not an exact string match) -- `LOG_LEVEL` as a name
promises graduated severity (`DEBUG`/`INFO`/`WARN`/...) this app has
never had; only one value (`OFF`) ever did anything, and everything
else, including a plausible `LOG_LEVEL=DEBUG` or `LOG_LEVEL=3`, was
silently ignored. `LOG_QUIET` is honest about being a boolean switch,
and the any-non-empty-value check matches the original
pre-`LOG_LEVEL` `LAMBADA_QUIET` semantics rather than requiring a
specific magic string.

`internal/envutil` was reconsidered for the same "is this worth its own
package" treatment and kept as-is: unlike `loglevel`, it's a genuinely
parameterized helper called with three different argument pairs across
two binaries (`scanDir`, `listenAddr`, `attachmentDir`), so one shared
copy is pulling real weight, not just avoiding two lines of duplication.

## Tooling: lint targets auto-fix, `check` stays report-only

`package.json`'s `lint-js`/`lint-go` now run with `--fix`
(`standard --fix`, `golangci-lint run --fix`) for local convenience.
`check` no longer calls through those scripts -- it invokes `standard`
and `golangci-lint run` directly, without `--fix`, so CI and `npm run
check` stay non-mutating. A new `format-go` script
(`go fmt ./...`) covers tabs-vs-spaces formatting, which golangci-lint's
default linter set (`errcheck`, `gosimple`, `govet`, `ineffassign`,
`staticcheck`, `unused`) never actually checked. `scandalous` picked up
the equivalent `lint-js`/`lint-rb` `--fix` treatment in parallel (no Go
side there, so no `format-go` equivalent).

`docs/FRAMEWORK.md` gained a new section this cycle too --
`## Narrow helpers worth pulling in for a future template` -- documenting
two real gaps found while asking "what would we not want to hand-roll
again for the next Go web project": `middleware.go`'s `withLogging`
doesn't forward optional `http.ResponseWriter` interfaces
(`http.Flusher`/`http.Hijacker`), which
[`httpsnoop`](https://github.com/felixge/httpsnoop) exists to fix, and
`listingTemplate.Execute` writing straight to the live response means a
mid-render failure leaves a partial `200` instead of a clean `500`,
which [`bpool`](https://github.com/oxtoacart/bpool)'s
buffer-then-flush pattern exists to fix. Neither pulled in yet -- purely
documented for if/when either gap actually bites.

## Upgrading

**Breaking config change:** `LOG_LEVEL=OFF` is no longer read by either
binary. If you have it set anywhere (a systemd override on the Pi, a
shell profile), switch it to `LOG_QUIET=1` -- same as the
`LAMBADA_QUIET` -> `LOG_LEVEL` migration in 2.8.0, this is a rename, not
new behavior, but the old variable name is silently ignored now rather
than erroring.

No dependency changes, no wire-format changes -- `/files.json`'s shape
is untouched. Behavior a client could actually observe: a
permission-denied `download`/`delete` now returns 500 instead of a
misleading 404, a directory-listing failure that used to 500 now renders
an empty listing (logged server-side) instead of failing the request
outright, and a failed `DELETE` (`os.Remove` error) still 500s but no
longer logs a dedicated line for it.

Confirmed via `npm run check` and CI on real hardware.
